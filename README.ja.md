## 概要

**Speckit**は、機能開発を体系的に管理するためのワークフローシステムです。「仕様策定」→「技術計画」→「タスク分解」→「実装」という明確なフェーズに分けることで、品質の高い開発を実現します。

### なぜSpeckitを使うのか？

- **明確な責任分離**: 「何を作るか(WHAT)」と「どう作るか(HOW)」を分離
- **段階的な詳細化**: 大まかな要求から具体的なタスクへ段階的に落とし込む
- **トレーサビリティ**: ユーザーストーリーから実装タスクまで追跡可能
- **品質保証**: 各フェーズでの検証により、手戻りを最小化

## クイックスタート

### 1. 機能仕様を作成

```bash
/speckit.specify ユーザー認証機能を追加したい
```

これにより:
- 機能用のブランチが作成される (例: `1-user-auth`)
- 仕様書 `specs/1-user-auth/spec.md` が生成される
- ビジネス価値とユーザーニーズに焦点を当てた仕様が作られる

### 2. 曖昧さを解消 (任意)

```bash
/speckit.clarify
```

AIが仕様の曖昧な部分を検出し、最大5つの質問で明確化します。

### 3. 技術計画を作成

```bash
/speckit.plan
```

仕様をもとに、技術スタック、アーキテクチャ、データモデルを決定します。

### 4. タスクに分解

```bash
/speckit.tasks
```

計画を実行可能なタスクリストに分解します。各タスクはユーザーストーリーごとに整理されます。

### 5. 実装を実行

```bash
/speckit.implement
```

タスクリストに従って、段階的に実装を進めます。

## 基本コンセプト

### ワークフローの全体像

```
┌─────────────────┐
│  機能のアイデア  │ ← ユーザーの要求・ビジネスニーズ
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ /speckit.specify│ ← 仕様策定 (WHAT/WHY)
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ /speckit.clarify│ ← 曖昧さの解消 (任意)
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  /speckit.plan  │ ← 技術計画 (HOW)
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ /speckit.tasks  │ ← タスク分解
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│/speckit.analyze │ ← 整合性検証 (任意)
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│/speckit.implement│ ← 実装実行
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│   完成した機能   │
└─────────────────┘
```

### 重要な原則

#### 1. WHAT と HOW の分離

- **spec.md (仕様書)**: ユーザーが何を必要としているか、なぜ必要か
  - ❌ 「FastAPIを使ってRESTful APIを作る」
  - ✅ 「ユーザーは2分以内にアカウント作成を完了できる」

- **plan.md (計画書)**: どの技術でどう実装するか
  - ✅ 「FastAPI + PostgreSQL + JWT認証」

#### 2. ユーザーストーリー中心

全ての機能は「ユーザーストーリー」として定義され、優先度付けされます:

- **P1 (最優先)**: MVP (Minimum Viable Product) に必須
- **P2 (高優先度)**: 重要だが後回し可能
- **P3 (中優先度)**: あると嬉しい機能

各ストーリーは独立してテスト・デプロイ可能です。

#### 3. 段階的な詳細化

```
大まかな要求
    ↓
機能仕様 (ビジネス視点)
    ↓
技術計画 (開発視点)
    ↓
タスクリスト (実行レベル)
    ↓
コード
```

各段階で詳細度を上げていくことで、早期に方向性を確認できます。

## ディレクトリ構造

### プロジェクトルート

```
my-project/
├── .specify/              # Speckitシステムファイル
│   ├── memory/
│   │   └── constitution.md    # プロジェクト憲章
│   ├── scripts/
│   │   └── bash/              # 自動化スクリプト
│   └── templates/             # テンプレート
├── .claude/
│   └── commands/              # スラッシュコマンド定義
├── specs/                     # 機能ごとのワークスペース
│   └── 1-user-auth/           # 例: ユーザー認証機能
│       ├── spec.md            # 機能仕様書
│       ├── plan.md            # 実装計画書
│       ├── tasks.md           # タスクリスト
│       ├── research.md        # 技術調査結果
│       ├── data-model.md      # データモデル
│       ├── contracts/         # API契約
│       └── checklists/        # 品質チェックリスト
└── src/                       # 実際のソースコード
    └── (実装されたコード)
```

### 各ファイルの役割

| ファイル | 作成コマンド | 内容 |
|---------|-------------|------|
| `spec.md` | `/speckit.specify` | 機能仕様 (ユーザー視点) |
| `plan.md` | `/speckit.plan` | 技術計画 (開発者視点) |
| `tasks.md` | `/speckit.tasks` | タスク分解 (実行レベル) |
| `research.md` | `/speckit.plan` | 技術選定の調査結果 |
| `data-model.md` | `/speckit.plan` | エンティティとリレーション |
| `contracts/` | `/speckit.plan` | API仕様 (OpenAPI等) |
| `checklists/` | `/speckit.checklist` | 品質チェック項目 |
